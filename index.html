1. Standard ISO/IEC 9075:2023

SQL/PGQ (Property Graph Queries)

Cosa fa: Le Property Graph Queries (introdotte nello standard SQL:2023) permettono di interrogare modelli relazionali standard come se fossero grafi composti da "Nodi" e "Archi" (relazioni).

A cosa serve: Evita di dover migrare i dati verso database NoSQL puri (come Neo4j) quando si ha bisogno di fare ricerche strutturali complesse, come trovare reti di frode finanziaria, percorsi di rete o raccomandazioni di prodotti.

-- Ricerca di percorsi nei grafi relazionali (Standard 2023)
-- Troviamo tutti i trasferimenti di denaro in un grafo finanziario
SELECT * FROM GRAPH_TABLE (
    Financial_Graph
    MATCH (a IS Account) -[t IS Transfer]-> (b IS Account)
    WHERE a.owner = 'Nikita Ivanov'
    COLUMNS (a.id AS from_id, b.id AS to_id, t.amount)
);

2. Intelligenza Artificiale & Tipi Vettoriali

Tipo Vettoriale e Ricerca di Similarità

Cosa fa: Il tipo di dato VECTOR memorizza array di numeri a virgola mobile (embeddings). La funzione VECTOR_DISTANCE() calcola la distanza matematica tra due vettori.

A cosa serve: Abilita la Semantic Search e la RAG direttamente nel database relazionale, trovando documenti simili "per concetto".

-- Creazione tabella con tipo di dato VECTOR
CREATE TABLE Documenti (
    doc_id INT PRIMARY KEY,
    contenuto NVARCHAR(MAX),
    embedding VECTOR(1536) 
);

-- Ricerca di Similarità (Coseno)
SELECT TOP 5 
    doc_id, 
    contenuto
FROM Documenti
ORDER BY VECTOR_DISTANCE('cosine', embedding, @query_vector);

3. Delta Lake & Architettura Lakehouse

Time Travel & Ottimizzazione (Photon)

Cosa fa: Il Time Travel interroga versioni precedenti dei dati. OPTIMIZE e VACUUM gestiscono lo storage fisico accelerato.

A cosa serve: Rollback rapido da errori accidentali, auditing, e pulizia automatica dello spazio su disco.

-- Interrogazione basata sul Timestamp (Time Travel)
SELECT * FROM target_table TIMESTAMP AS OF '2026-03-28 14:30:00';

-- Rollback della tabella per annullare una DELETE accidentale
RESTORE TABLE target_table TO VERSION AS OF 5;

-- Ottimizzazione fisica: raggruppa dati simili
OPTIMIZE target_table ZORDER BY (customer_id, order_date);

-- Pulizia logica: Rimuove i file fisici obsoleti
VACUUM target_table RETAIN 168 HOURS;

4. Operazioni DML: Il comando MERGE

MERGE INTO (Upsert Pattern)

Cosa fa: MERGE INTO valuta una condizione tra una tabella di origine (dati nuovi) e una di destinazione (dati vecchi). A seconda dell'esito, esegue in un solo passaggio un'azione di INSERT, UPDATE o DELETE.

A cosa serve: È il cuore del Data Engineering (processi ETL/ELT). Serve per sincronizzare enormi tabelle quotidianamente: invece di svuotare e ricaricare milioni di righe (lento), si aggiornano solo i record modificati e si inseriscono solo i nuovi (Upsert).

-- Pattern UPSERT (Update o Insert) Atomico
MERGE INTO customers AS target
USING updates AS source
ON target.customer_id = source.customer_id

-- Se il cliente esiste già ed è attivo, aggiorna email e data
WHEN MATCHED AND source.status = 'active' THEN
    UPDATE SET target.email = source.email, target.updated_at = current_timestamp()

-- Se il cliente esiste ma nella sorgente è marcato come cancellato, eliminalo
WHEN MATCHED AND source.status = 'deleted' THEN
    DELETE

-- Se il cliente NON esiste nella tabella di destinazione, crea la riga
WHEN NOT MATCHED THEN
    INSERT (customer_id, email, status) 
    VALUES (source.customer_id, source.email, source.status);

5. Funzioni: Aggregazione & Window (Analitiche)

Funzioni Aggregate: COUNT, SUM, AVG, LISTAGG

Cosa fanno: Elaborano un insieme di righe (create tramite GROUP BY) e restituiscono un singolo valore riassuntivo per gruppo. Nota: ignorano sempre i valori NULL (eccetto COUNT(*)).

  • COUNT / SUM / AVG / MIN / MAX: Contano le righe, sommano valori, estraggono la media, il minimo o il massimo. Servono per metriche base (KPIs).
  • LISTAGG (o STRING_AGG): Prende i valori testuali di più righe e li concatena in una singola stringa separata da virgole. Uso: raggruppare i nomi dei dipendenti per dipartimento in un'unica cella.
SELECT 
    department,
    COUNT(employee_id) AS total_emps,
    AVG(salary) AS avg_salary,
    -- Concatena in una stringa i nomi: "Mario, Luigi, Anna"
    LISTAGG(employee_name, ', ') WITHIN GROUP (ORDER BY employee_name) AS emp_list
FROM employees
GROUP BY department;
Funzione Window: NTILE (Creare Fasce/Categorie)

Cosa fa: Immagina di prendere una classifica (dal più alto al più basso) e di volerla tagliare in "fette" uguali. NTILE(n) fa esattamente questo: divide i tuoi risultati in n gruppi e assegna a ogni riga l'etichetta del suo gruppo (Es: Gruppo 1, Gruppo 2, Gruppo 3).

A cosa serve: È la funzione perfetta per il marketing e la segmentazione! Vuoi dividere automaticamente i tuoi clienti in "Clienti VIP", "Clienti Medi" e "Clienti Base" in base a quanto spendono? Usa NTILE(3).

-- Suddivisione dei clienti in 3 fasce di spesa (Es: 1=VIP, 2=Medi, 3=Base)
SELECT 
    customer_name,
    total_spent,
    -- Ordina per spesa (dal più alto al più basso) e taglia in 3 gruppi
    NTILE(3) OVER (ORDER BY total_spent DESC) as tier_cliente
FROM customer_sales;
Funzioni Window: ROW_NUMBER, LEAD, LAG

Cosa fanno: Eseguono calcoli su un set di righe correlate alla riga corrente (la "finestra" creata da OVER()), senza raggruppare le righe in un singolo output.

  • ROW_NUMBER: Assegna un ID sequenziale a ogni riga dentro la sua partizione (es. 1,2,3). Uso: Top-N Queries (trovare i 3 dipendenti più pagati per ogni reparto).
  • LEAD / LAG: Accedono ai dati della riga "successiva" (LEAD) o "precedente" (LAG). Uso: calcolo della differenza mese-su-mese o rilevamento trend.
SELECT 
    department,
    employee_name,
    salary,
    -- Classifica assoluta all'interno del dipartimento basata sullo stipendio
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) as rank_stipendio,
    
    -- Prende lo stipendio di chi è stato assunto PRIMA (LAG) e DOPO (LEAD)
    LAG(salary) OVER (PARTITION BY department ORDER BY hire_date) as prev_hired_salary,
    LEAD(salary) OVER (PARTITION BY department ORDER BY hire_date) as next_hired_salary
FROM employees;

6. Funzioni Scalari, JSON, Date e Geo

Matematiche e Stringhe (GREATEST, ABS, UPPER, LPAD)

Cosa fanno: Le funzioni scalari prendono uno o più parametri in input per una singola riga e restituiscono sempre e solo un singolo valore formattato/calcolato.

  • GREATEST / LEAST: Diversamente da MIN/MAX, queste operano sulla stessa riga tra colonne diverse. Restituiscono il massimo o il minimo tra una lista di espressioni.
  • ABS / MOD / ROUND: ABS rimuove il segno negativo (valore assoluto). MOD calcola il resto di una divisione. ROUND arrotonda i decimali.
  • UPPER / LTRIM / RTRIM / LPAD: UPPER converte in maiuscolo. LTRIM/RTRIM rimuovono spazi bianchi. LPAD aggiunge caratteri di riempimento a sinistra (padding).
SELECT 
    -- Matematica: Qual è il valore più alto tra Q1, Q2 e Q3 per questo specifico record?
    GREATEST(sales_q1, sales_q2, sales_q3) AS max_quarter_sales,
    ABS(accounting_balance) AS absolute_balance, -- Rimuove segno negativo
    
    -- Stringhe: formatta e pulisci nomi ed IDs
    UPPER(LTRIM(RTRIM(customer_name))) AS clean_name,
    LPAD(customer_id, 10, '0') AS padded_id -- Risultato: '0000000123'
FROM financial_data;
Gestione JSON, Date e Dati Geospaziali

Il database relazionale moderno deve saper navigare tra formati non strutturati, coordinate geografiche e calcoli temporali deterministici.

  • JSON_VALUE / Dot Notation: Estrae uno specifico valore scalare (es. una stringa o un numero) da una colonna testuale che contiene un payload JSON.
  • GETDATE / current_timestamp(): Funzioni "non-deterministiche" che restituiscono l'ora attuale. DATEADD calcola scostamenti temporali (es. aggiungere 30 giorni).
-- 1. Estrazione JSON (Metodo classico SQL/JSON vs Dot Notation moderna)
SELECT 
    JSON_VALUE(payload, '$.user.address.city') AS json_city,
    payload.user.address.city AS dot_notation_city -- (Stesso risultato, standard 2026)
FROM api_logs;

-- 2. Gestione Date
SELECT 
    order_id,
    current_timestamp() AS execution_time,
    DATEADD(day, 30, order_date) AS payment_due_date -- Aggiunge 30 giorni
FROM orders;

7. Comandi Base & Filtri (CRUD)

Filtraggio Dati: WHERE, LIKE, IN e BETWEEN

Cosa fanno: Questi comandi formano l'ossatura di ogni interrogazione SQL (lettura dei dati). Permettono di estrarre colonne specifiche (SELECT), filtrare le righe in base a condizioni (WHERE), e ordinare l'output (ORDER BY).

  • WHERE: Il filtro principale. Applica operatori logici (=, !=, >, <, AND, OR).
  • LIKE: Esegue il "Pattern Matching" sulle stringhe. Usando il jolly % puoi cercare parole che "iniziano per" o "contengono" una certa frase.
  • IN / BETWEEN: IN verifica se un valore è contenuto in una lista specifica (es. stato IN ('IT', 'FR')). BETWEEN controlla se un numero o data cade in un intervallo chiuso.
-- Selezione mirata di utenti attivi con domini email specifici
SELECT 
    nome, 
    cognome, 
    email,
    data_registrazione
FROM utenti
WHERE nazione = 'IT' 
  AND status = 'Attivo'
  AND (email LIKE '%@gmail.com' OR email LIKE '%@yahoo.com')
  AND data_registrazione BETWEEN '2024-01-01' AND '2025-12-31'
ORDER BY cognome ASC, data_registrazione DESC
LIMIT 100;

8. JOIN (Relazioni tra Tabelle)

INNER JOIN (Intersezione Rigorosa)

Cosa fa: L'INNER JOIN è la join di default. Restituisce esclusivamente i record che trovano una corrispondenza esatta in entrambe le tabelle collegate dalla condizione (di solito Primary Key = Foreign Key).

A cosa serve: Si usa quando ti interessano solo i dati completi. Ad esempio, "Fammi vedere solo i clienti che hanno effettivamente piazzato almeno un ordine".

-- Esempio: INNER JOIN
SELECT 
    c.nome_cliente,
    o.id_ordine,
    o.importo
FROM clienti AS c
INNER JOIN ordini AS o 
    ON c.id_cliente = o.id_cliente;
LEFT JOIN (Mantenimento a Sinistra)

Cosa fa: La LEFT JOIN (o Left Outer Join) garantisce che tutti i record della tabella di sinistra vengano restituiti, indipendentemente dal fatto che abbiano o meno un legame a destra. In caso di assenza, le colonne di destra saranno valorizzate a NULL.

A cosa serve: Fondamentale nell'analisi esplorativa. Ad esempio: "Mostrami tutto il mio database clienti, associandogli gli ordini se esistono, altrimenti lasciando il campo ordine vuoto".

-- Esempio: LEFT JOIN
SELECT 
    c.nome_cliente,
    o.id_ordine,
    o.importo
FROM clienti AS c
LEFT JOIN ordini AS o 
    ON c.id_cliente = o.id_cliente;
FULL OUTER JOIN (Unione Totale)

Cosa fa: La FULL OUTER JOIN combina il comportamento di una Left e Right Join. Mantiene tutte le righe di entrambe le tabelle. Se manca la corrispondenza da un lato, inserisce un NULL.

A cosa serve: Indispensabile per riconciliazioni contabili o Quality Check. "Voglio vedere tutto: sia i clienti senza ordini, sia eventuali ordini rimasti 'orfani' associati a clienti cancellati dal sistema".

-- Esempio: FULL OUTER JOIN
SELECT 
    c.nome_cliente,
    o.id_ordine,
    o.importo
FROM clienti AS c
FULL OUTER JOIN ordini AS o 
    ON c.id_cliente = o.id_cliente;
LEFT SEMI JOIN (Filtro di Esistenza Efficiente)

Cosa fa: Un costrutto nativo (molto diffuso in Databricks/Spark) che agisce come un EXISTS. Restituisce solo le righe della tabella sinistra che hanno almeno un match in quella di destra. Cruciale: non duplica mai le righe di sinistra (anche se ci sono match multipli a destra) e ignora tutte le colonne della tabella destra.

A cosa serve: Sicurezza e prestazioni. "Fammi l'elenco dei soli clienti che hanno acquistato". Con un'Inner Join otterresti un cliente ripetuto 10 volte se ha 10 ordini; con la Semi Join lo ottieni esattamente 1 volta. Ideale per estrarre target list pulite.

-- Esempio: LEFT SEMI JOIN
-- Ritorna solo i dati del cliente, senza duplicarli se ci sono ordini multipli
SELECT 
    c.nome_cliente
FROM clienti AS c
LEFT SEMI JOIN ordini AS o 
    ON c.id_cliente = o.id_cliente;
LEFT ANTI JOIN (Ricerca Orfani / Not Exists)

Cosa fa: L'opposto della Semi Join. Si comporta come un super-efficiente NOT EXISTS. Restituisce solo i record della tabella sinistra che non hanno alcuna corrispondenza in quella di destra.

A cosa serve: Individuazione anomalie, frodi o pulizia dati. "Dammi la lista esatta di tutti gli utenti registrati che non hanno MAI fatto un ordine". Evita di dover scrivere complesse LEFT JOIN con filtri WHERE id IS NULL, offrendo un piano di esecuzione nativo e velocissimo.

-- Esempio: LEFT ANTI JOIN
-- Intercetta immediatamente gli utenti inattivi
SELECT 
    c.nome_cliente
FROM clienti AS c
LEFT ANTI JOIN ordini AS o 
    ON c.id_cliente = o.id_cliente;

9. CTE (Common Table Expressions) & Subquery

Strutturare query complesse con la clausola WITH

Cosa fa: Una CTE (Common Table Expression), introdotta dalla parola chiave WITH, ti permette di definire una tabella virtuale temporanea che esiste solo durante l'esecuzione della query corrente.

A cosa serve: Sostituisce le vecchie "Subqueries Nidificate" (query dentro una query) rendendo il codice SQL lungo e complesso molto più leggibile, organizzato in blocchi logici e facile da mantenere o debuggare. È utilissima per eseguire calcoli aggregati in step successivi.

-- Step 1: Creiamo una tabella temporanea (CTE) con le vendite aggregate per agente
WITH VenditeAgente AS (
    SELECT 
        id_agente,
        COUNT(id_ordine) AS totale_ordini,
        SUM(importo) AS fatturato_totale
    FROM ordini
    WHERE data_ordine >= '2025-01-01'
    GROUP BY id_agente
)

-- Step 2: Usiamo la CTE appena creata e la mettiamo in JOIN con l'anagrafica
SELECT 
    a.nome_agente,
    v.totale_ordini,
    v.fatturato_totale
FROM agenti AS a
INNER JOIN VenditeAgente AS v 
    ON a.id_agente = v.id_agente
WHERE v.fatturato_totale > 100000.00 -- Filtriamo solo i Top Performers
ORDER BY v.fatturato_totale DESC;
